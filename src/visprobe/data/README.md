# RobustBench leaderboard snapshots

Frozen JSON snapshots of the RobustBench leaderboard. Read at runtime by `visprobe.leaderboard.RobustBenchClient`. Regenerated by `scripts/refresh_leaderboard.py` and **automatically refreshed weekly** by `.github/workflows/refresh-leaderboard.yml` — Mondays 06:00 UTC. The workflow opens a PR only when the leaderboard actually moved (the refresher preserves `snapshot_date` when entries are unchanged, so a spurious date-only PR cannot happen).

## File naming

`robustbench_<dataset>_<threat>.json` — one file per `(dataset, threat)` pair.

Currently shipped:
- `robustbench_cifar10_linf.json` — CIFAR-10 Linf (eps=8/255)
- `robustbench_imagenet_linf.json` — ImageNet Linf (eps=4/255)

Files ending in `.sample.json` are hand-curated five-entry samples used during early v3 development (milestone M1). They are deleted and replaced by real refresher output once `scripts/refresh_leaderboard.py` lands in M2. Do **not** rely on `.sample.json` files in shipped code.

## Top-level schema (v1)

| Field | Type | Required | Description |
|---|---|---|---|
| `schema_version` | int | yes | Must be `1`. `RobustBenchClient` raises on unknown future versions. Bump only when entry-field structure changes incompatibly. |
| `dataset` | string | yes | `"cifar10"` or `"imagenet"`. |
| `threat` | string | yes | Threat model name, e.g. `"Linf"`. |
| `eps` | float | yes | Attack budget at this threat model. CIFAR-10 Linf uses `8/255`; ImageNet Linf uses `4/255`. |
| `snapshot_date` | string | yes | ISO date (`YYYY-MM-DD`) when the snapshot was generated. Printed in every `LeaderboardComparison` render so staleness is visible. |
| `entries` | array | yes | Leaderboard entries, sorted by `robust_acc` descending. |

Additional top-level fields (e.g., `note` in the sample files) are **ignored** by the parser. This is intentional — forward-compatible JSON. Don't rely on unknown fields existing.

## Entry schema

Each item in `entries`:

| Field | Type | Required by parser | Description |
|---|---|---|---|
| `rank` | int | yes | Position in the leaderboard, 1-indexed. |
| `name` | string | yes | RobustBench model identifier. This is the value `robustbench.utils.load_model` expects for `model_name=`. |
| `clean_acc` | float in [0, 1] | yes | Top-1 accuracy on the clean test set. |
| `robust_acc` | float in [0, 1] | yes | Robust accuracy under full AutoAttack at `eps`. |
| `paper` | string | no (display only) | Paper title for the leaderboard render. |
| `venue` | string | no (display only) | Conference + year for the render. |
| `architecture` | string | no (display only) | Architecture description for the render. |

**Validation.** `RobustBenchClient.__init__` validates every entry on load: missing required fields, `null` values, non-numeric `clean_acc`/`robust_acc`, or floats outside `[0, 1]` raise `ValueError` with the bad field named. Display-only fields are allowed to be missing (rendered as `""`).

## Refresh cadence

The CI workflow `.github/workflows/refresh-leaderboard.yml` runs every Monday at 06:00 UTC. It calls `scripts/refresh_leaderboard.py` and, if any `src/visprobe/data/*.json` actually moved, opens a PR titled `chore: refresh RobustBench leaderboard snapshot`. A maintainer reviews and merges.

The workflow can also be triggered manually from the GitHub Actions tab via `workflow_dispatch` — useful for one-off mid-week refreshes.

The refresher preserves `snapshot_date` when entries are byte-identical to the prior snapshot. A spurious date-only PR cannot happen — PRs land only when the upstream leaderboard actually moved.

## Why a snapshot, not a live fetch

We considered fetching the leaderboard from RobustBench's GitHub at runtime. Rejected because:
- Network dependency at the worst time (analysis often happens offline)
- Mandatory caching layer with its own invalidation logic
- Non-reproducibility (same code, different rank two weeks later)
- No deterministic tests without mocking
- Rate limits on unauthenticated GitHub requests

Snapshots are cheap to refresh (CI runs the script weekly and opens a PR with the diff), and `snapshot_date` is shown in every comparison output so users always see how fresh the data is.
